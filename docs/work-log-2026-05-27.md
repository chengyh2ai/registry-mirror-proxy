# 2026-05-27 工作记录：Docker Registry Mirror Proxy

## 背景

客户环境访问 Docker Hub 官方地址不稳定，直接拉取镜像时经常出现：

```text
Get "https://registry-1.docker.io/v2/": context deadline exceeded
```

已有可用上游镜像地址：

```text
https://<upstream-registry-domain>
```

但不希望客户看到该域名。因此实现一个代理程序，让客户只配置：

```json
{
  "registry-mirrors": ["https://192.168.44.100"]
}
```

代理服务器再去访问火山引擎镜像仓库。

### 2. Go 项目实现

初始化 Go 项目，并实现以下模块：

```text
cmd/registry-mirror-proxy/main.go
internal/config/
internal/proxy/
internal/auth/
internal/logging/
internal/server/
configs/config.example.yaml
deploy/systemd/registry-mirror-proxy.service
```

核心能力：

- Docker Registry V2 拉取代理。
- `/v2/` 本地返回 `200 OK`，避免 Docker daemon 判断 mirror 不可用后回退 Docker Hub。
- Manifest、blob、tag list 等路径转发到上游。
- 默认只允许 `GET`、`HEAD`、`OPTIONS`。
- 默认拒绝 `POST`、`PUT`、`PATCH`、`DELETE`。
- 流式转发大镜像层，避免完整镜像层进入内存。
- 服务端内部跟随 HTTPS 重定向，避免把上游 `Location` 暴露给客户。
- 清洗响应头，隐藏上游域名。
- 清洗错误响应，避免泄露上游域名。
- 健康检查：`/healthz`。
- 就绪检查：`/readyz`。
- Prometheus 风格 metrics：`/metrics`。
- 可选 CIDR 客户端白名单。
- 可选并发控制。
- 可选磁盘 blob cache。
- 多上游故障切换。

### 3. 火山引擎临时访问密钥支持

根据火山引擎文档：

```text
GetAuthorizationToken
https://www.volcengine.com/docs/6420/113753?lang=zh
```

新增自动获取临时访问密钥能力：

- 代理可以通过火山 AK/SK 调用 `GetAuthorizationToken`。
- 自动获取 `Username` 和 `Token`。
- 缓存临时凭据。
- 在过期前自动刷新。
- 用返回的 `Username/Token` 作为上游 Basic Auth。
- 客户侧无需执行火山引擎的 `docker login`。
- 客户侧也不需要知道真实上游域名。

新增配置项：

```yaml
upstream_auth_enabled: true
upstream_access_key: ""
upstream_secret_key: ""
upstream_region: "cn-beijing"
upstream_endpoint: "<upstream-auth-endpoint>"
upstream_registry: ""
upstream_refresh_before: 10m
```

也支持 systemd 环境变量：

```ini
Environment="REGISTRY_MIRROR_UPSTREAM_AUTH_ENABLED=true"
Environment="REGISTRY_MIRROR_UPSTREAM_ACCESS_KEY=your-ak"
Environment="REGISTRY_MIRROR_UPSTREAM_SECRET_KEY=your-sk"
Environment="REGISTRY_MIRROR_UPSTREAM_REGION=cn-beijing"
Environment="REGISTRY_MIRROR_UPSTREAM_ENDPOINT=<upstream-auth-endpoint>"
Environment="REGISTRY_MIRROR_UPSTREAM_REGISTRY=your-registry-name"
```

注意：`REGISTRY_MIRROR_UPSTREAM_REGISTRY` 是火山镜像仓库实例名称，不是镜像仓库访问域名。

### 5. 排障与修复记录

#### 问题 1：systemd 服务启动失败

现象：

```text
registry-mirror-proxy.service failed
```

原因：

- 服务使用 `User=registry-mirror` 非 root 用户运行。
- 监听 `:443` 需要 `CAP_NET_BIND_SERVICE`。

修复：

```ini
AmbientCapabilities=CAP_NET_BIND_SERVICE
CapabilityBoundingSet=CAP_NET_BIND_SERVICE
```

同时发现旧 systemd 不识别：

```ini
ReadWritePaths=/var/cache/registry-mirror-proxy
```

已从模板中移除。

#### 问题 2：TLS bad certificate

现象：

```text
TLS handshake error ... remote error: tls: bad certificate
```

原因：

- Docker 客户端不信任代理证书，或证书没有包含 `192.168.44.100` 的 IP SAN。

临时处理：

客户侧使用：

```json
{
  "registry-mirrors": ["https://192.168.44.100"],
  "insecure-registries": ["192.168.44.100"]
}
```

生产建议：

- 使用被客户机器信任的 CA。
- 证书 SAN 必须包含：

```text
IP Address:192.168.44.100
```

#### 问题 3：Docker 回退访问 registry-1.docker.io

现象：

```text
Get "https://registry-1.docker.io/v2/": context deadline exceeded
```

代理日志显示：

```text
GET /v2/ status=401
```

原因：

- 代理把 `/v2/` 探活转发给上游。
- 上游返回 `401`。
- Docker daemon 判断 mirror 不可用，回退到 Docker Hub 官方地址。

修复：

- `/v2/` 和 `/v2` 由代理本地直接返回 `200 OK`。
- 响应头增加：

```text
Docker-Distribution-API-Version: registry/2.0
```

#### 问题 4：manifest 请求返回 401

现象：

```text
HEAD /v2/library/mysql/manifests/latest status=401
GET /v2/library/mysql/manifests/latest status=401
```

响应头包含：

```text
WWW-Authenticate: Bearer realm="https://192.168.44.100/service/token",service="harbor-registry",scope="repository:library/mysql:pull"
```

处理过程：

- 先实现 `WWW-Authenticate realm` 改写。
- 后发现 Docker 没有继续请求 `/service/token`。
- 最终改为代理内部处理认证：
  - 上游返回 401。
  - 代理解析 `WWW-Authenticate`。
  - 代理请求上游 token endpoint。
  - 代理带 `Authorization: Bearer ...` 重试原 manifest/blob 请求。

#### 问题 5：上游临时 token 手工维护不方便

处理：

- 接入火山官方 Go SDK。
- 使用 `GetAuthorizationToken` 自动获取临时访问密钥。
- 自动缓存并刷新。

## 当前部署建议

客户侧 Docker 配置：

```json
{
  "registry-mirrors": ["https://192.168.44.100"],
  "insecure-registries": ["192.168.44.100"]
}
```

生产建议替换为可信证书后移除 `insecure-registries`。

代理侧推荐使用 systemd drop-in 配置火山 AK/SK：

```bash
sudo systemctl edit registry-mirror-proxy
```

```ini
[Service]
Environment="REGISTRY_MIRROR_UPSTREAM_AUTH_ENABLED=true"
Environment="REGISTRY_MIRROR_UPSTREAM_ACCESS_KEY=your-ak"
Environment="REGISTRY_MIRROR_UPSTREAM_SECRET_KEY=your-sk"
Environment="REGISTRY_MIRROR_UPSTREAM_REGION=cn-beijing"
Environment="REGISTRY_MIRROR_UPSTREAM_ENDPOINT=<upstream-auth-endpoint>"
Environment="REGISTRY_MIRROR_UPSTREAM_REGISTRY=your-registry-name"
```

重启：

```bash
sudo systemctl daemon-reload
sudo systemctl restart registry-mirror-proxy
```

查看日志：

```bash
sudo journalctl -u registry-mirror-proxy -f -o cat
```

## 关于缓存

当前磁盘缓存默认关闭：

```yaml
enable_disk_cache: false
```

因此 `/var/cache/registry-mirror-proxy` 为空是正常现象。

如需开启：

```yaml
enable_disk_cache: true
disk_cache_dir: "/var/cache/registry-mirror-proxy"
disk_cache_max_bytes: 0
```

并确保目录可写：

```bash
sudo mkdir -p /var/cache/registry-mirror-proxy
sudo chown -R registry-mirror:registry-mirror /var/cache/registry-mirror-proxy
sudo systemctl restart registry-mirror-proxy
```

当前缓存策略：

- 只缓存完整 blob 请求。
- 不缓存 manifest。
- 不缓存 token。
- 不缓存带 `Range` 的分段请求。

## 安全注意事项

1. 今天聊天中曾贴出火山临时 token，建议在火山引擎侧吊销或等待其过期后重新生成。
2. AK/SK 不要写入客户机器。
3. AK/SK 不建议写入全局可读配置文件。
4. 推荐通过 systemd drop-in 或权限 `0600` 的配置文件保存敏感信息。
5. 生产环境应使用可信 CA 签发的 TLS 证书，并包含 IP SAN。
